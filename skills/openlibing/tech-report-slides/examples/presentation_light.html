omni-performance-improvement Skill

性能优化 4 阶段 Playbook

Phase 1 · Baseline

  • SQL 设计 → E2E 验证算子命中 → 执行时间调校
  • Native / Omni 各跑 5 次（1 冷 + 4 热），取均值
  • TPC-DS store_sales 约 2.87 亿行大表

Phase 2 · Profiling

  • async-profiler 火焰图（Omni + Native 各一份）
  • 执行计划抓取（算子耗时 / Stage 分布）
  • Stage 级 Duration / GC / Shuffle 分析

Phase 3 · Diagnose

  • 计算热点 / 内存分配 / null 低效 / 倾斜 / 锁竞争
  • 主瓶颈 + 次瓶颈，每个结论必须有证据支撑
  • 禁止无画像直接修改代码

Phase 4 · Optimize + Validate

  • 单一瓶颈对应优化方案，同步补充 UT
  • 优化后正确性 E2E 回归（本轮新增）
  • 三方对比（Native / Omni-Before / Omni-After）
  • 可回退，有结论，沉淀 case-study

能力验证

自迭代能力的四个具体体现

🔁

失败自愈

compile_omni 失败 → 读编译错误 → 定位问题 → 修改代码 → 重编译；q3 运行期报 MakeDecimal not found → 定位 vector path 缺失 → 三轮递进修复，全程无人工介入。

📈

证据驱动决策

每次性能优化前必须先采集火焰图。Opt3 关键突破（JNI 调用 1000→1 次）来自火焰图中 GetOutput 热点识别，而非猜测。硬性规则：没有画像不允许改代码。

📄

跨轮次记忆

每轮迭代的中间数据、当前 ratio、下一步方向全部通过文档持久化（modification_trace、perf report）。不依赖上下文窗口，状态显式管理，轮次间无信息丢失。

🔒

卡点有效性

Skill 中的强制文档卡点被实际遵守：context_sortagg.md 与 self_validation_sortagg.md 先于任何实现代码存在且内容完整。卡点不是建议，是约束。
