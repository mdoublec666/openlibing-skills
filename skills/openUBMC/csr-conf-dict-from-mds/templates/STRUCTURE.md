# CSR 配置字典 object 文档 — 公共结构归纳

本文档归纳 `docs/.../csr_conf_dict/object/*.md`（约 116 篇）中的**稳定共性**，供 `csr-conf-dict-from-mds` 模板与生成逻辑对齐。分析基线包含：`Sensor.md`、`CPU.md`、`Time.md`、`Scanner.md`、`VRDTypeMap.md`、`CPUDie.md` 及目录内 grep 抽样。

## 1. 稳定骨架（高一致）

| 顺序 | 一级标题（emoji） | 子结构 | 说明 |
|------|------------------|--------|------|
| 1 | `# 📋 文档信息` | 表格「项目 / 内容」 | 固定行：**文档标题、版本、创建日期、最后更新、维护状态**；维护状态多为 `✅ 活跃维护` |
| 2 | `# 📋 变更历史记录` | `## 文档变更记录` + 表格 | 列：**版本、发布日期、变更类型、变更内容、影响范围、维护人员**；少数文件 H1 写「变更历史」无「记录」，模板统一为 **变更历史记录** |
| 3 | `# 🎯 类概览` | `## …` + 表格「属性 / 值」 | 固定行：**类名称、功能描述、所属SIG组、所属组件、责任人、最后更新、状态**；**所属SIG组** 仅填 `component-sig-mapping.json` 中 `component_to_sig` 的值或 `default_sig`，**禁止**沿用旧文档中的业务组名（如 `system`、`sensor` 等与 OpenUBMC SIG 不一致的写法）；状态多为 `🟢 正常运行` |
| 4 | `# 📊 属性定义详表` | 多级 `## …必选属性` / `## …可选属性` | 表头 10 列一致；单类常用 `## 必选属性`，多类合并用 `## Entity 类 - 必选属性` 形式 |

## 2. 扩展章节（按题材取舍）

| 顺序 | 一级标题 | 典型内容 | 出现规律 |
|------|----------|----------|----------|
| 5 | `# 🔗 动态关联机制` | `## 语法规范` + 代码块 + `## 关联说明`（列表）；复杂题材可加多级 `##`（如「实际示例」「槽位变量替换」） | **大部份**完整版文档具备；**极简**文档（如仅少量属性、无关联语法）可整节省略 |
| 6 | `# 📂 分类标准` | `## 硬件属性` / `## 软件属性`（定义、特点、示例列表） | 与「动态关联」类似，完整版常见；极简篇可省略 |
| 7 | `# 📝 配置示例` | 一个或多个 `## 小节标题` + ` ```json ` 块 | **极常见**；可多条示例 |
| 8 | `# 🔧 使用指南` | 至少 `## 配置步骤`、`## 注意事项`、`## 故障排查`；`## 性能建议` **仅在有实质内容时**通过 `{PerformanceSectionOptional}` 插入 | **极常见** |
| 9 | `# 📚 相关文档` | 链接列表 | **少数**文档具备，默认整节省略 |

## 3. 版式细节（公共）

- 段间分隔：单独一行 `---`。
- Frontmatter：`title: CSR配置字典之{类}类`，`date: YYYY/M/D`（月日可单数字，与现网一致）。
- 类概览表第二列表格行末常见一个空格再 `| `（与现有一致即可，非强制）。
- 属性表：**属性名**列使用反引号 `` `Name` ``。
- 代码块：语法规范多为 `yaml` 或 `json`；配置示例为 `json`。

## 4. 与 `model.json` 生成策略的对应关系

- **必选四块**：文档信息、变更历史、类概览、属性详表（与产品需求一致）。
- **扩展块**：是否生成「动态关联」「分类标准」建议由规则判定：详表 CSR 属性中出现 `<=/`、`#/`、`expr`、`${` 等任一模式 → 保留「动态关联」占位或生成简述；否则可删除该节。「分类标准」在能归纳硬件/软件示例列表时保留，否则可删或缩成一段说明。
- **配置示例 / 使用指南**：优先保留骨架；内容不足用占位，禁止长篇杜撰。

## 5. 模板文件职责与拼接顺序

为减少文件数、保持拼装效果不变，`templates/` 下用于拼装的 Markdown 共 **4 个**（本文件 `STRUCTURE.md` 仅为说明，不参与拼装）：

| 文件 | 职责 |
|------|------|
| `document_head.md` | YAML frontmatter + 文档信息 → 变更历史 → 类概览 → `# 📊 属性定义详表` + `{AttributeBlocks}` |
| `section_attributes_block.md` | 单类「必选 / 可选」两表；多类则重复多份并替换 `{InnerClassName}` |
| `fragment_linkage_classification.md` | **可选**：动态关联 + 分类标准（有材料时接在属性表之后） |
| `fragment_suffix_examples_guide.md` | 配置示例 + 使用指南 + 可选 `{PerformanceSectionOptional}` + `{RelatedDocSectionOptional}` |

全篇排版参考见仓库内 `examples/complete_example.md`（与上述片段拼装结果等价）。

**推荐拼接顺序**：`document_head.md`（`{AttributeBlocks}` 已展开）→（可选）`fragment_linkage_classification.md` → `fragment_suffix_examples_guide.md`

## 6. 属性表表头与分隔行（常量）

以下两行与 `section_attributes_block.md` 中一致，勿增删列（10 列）：

```
| 属性名 | 类型 | 默认值 | 取值范围 | 动态关联 | 描述 | 使用场景 | 举例 | 来源 | 分类 |
|--------|------|--------|----------|----------|------|----------|------|------|------|
```

## 7. 占位符约定（生成时替换）

| 占位符 | 含义 |
|--------|------|
| `{ClassOverviewH2}` | 类概览二级标题全文，如 `Time 类`、`Sensor 类系统` |
| `{ClassNameDisplay}` | 类名称列展示，可与 `{ClassName}` 相同，或合并多类时写 `` `Sensor` (Entity + …) `` |
| `{AttributeBlocks}` | 一份或多份 `section_attributes_block.md` 拼接结果 |
| `{LinkageCodeFence}` | `yaml` 或 `json` 等，与现网示例一致 |
| `{ConfigExampleBlocks}` | 一个或多个 `## 小标题` + 代码块；极简时可一节 + 占位 JSON |
| `{PerformanceSectionOptional}` | **有**性能/调优要点时替换为完整 `## 性能建议` 小节（含标题与列表）；**无**则替换为**空字符串**（不要保留空标题；禁止写「无」占位） |
| `{RelatedDocSectionOptional}` | **有**外链时替换为：`---` + 换行 + `# 📚 相关文档` + 换行 + 列表；**无**则空字符串 |
