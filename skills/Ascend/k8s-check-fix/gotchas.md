# 错题本（Gotchas）

本文档记录了 Kubernetes 诊断中常见的错误做法和正确思路，帮助模型避免踩坑。在执行诊断或给出建议前，建议快速浏览相关条目。

---

## 1. Pod 相关陷阱

### 1.1 CrashLoopBackOff
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 只查看当前日志，未查看上一次容器日志 | 崩溃后容器重启，当前日志通常是启动成功后的输出，错误信息在上一次日志中 |
| 直接删除 Pod 认为问题解决 | 如果 Pod 由 Deployment 管理，删除后会自动重建，不解决根本原因。应分析日志、事件、资源限制等 |
| 只关注重启次数，忽略 OOMKilled 终止原因 | 查看 `state.terminated.reason`，若为 `OOMKilled` 则需增加内存 limit 或优化应用 |

### 1.2 ImagePullBackOff / ErrImagePull
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 仅检查镜像名称拼写 | 还需检查镜像仓库认证（imagePullSecrets）、网络连通性、私有仓库证书 |
| 直接删除 Pod 尝试重新拉取 | 应先确认镜像是否存在、标签是否正确、Secret 是否配置 |

### 1.3 Pending（待调度）
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 仅检查节点资源是否充足 | 还应检查：节点污点与容忍度、节点亲和性、PVC 是否绑定、调度器事件 |
| 假设是资源不足就盲目增加节点 | 应先检查事件中的 `FailedScheduling` 具体原因，可能因持久卷未绑定或节点选择器不匹配 |

### 1.4 容器运行但未就绪（Ready = False）
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 认为容器启动失败 | 容器可能正常运行但健康检查（readinessProbe）失败，应检查探针配置和容器内部服务状态 |

## 2. 节点相关陷阱

### 2.1 节点 NotReady
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 直接驱逐节点或删除节点 | 应先登录节点检查 kubelet 状态、证书有效期、网络连通性、系统资源 |
| 忽略节点条件中的压力标志 | 如果 `MemoryPressure` 或 `DiskPressure` 为 True，应先清理资源，再考虑重启 kubelet |

### 2.2 节点资源压力
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 立即增加节点规格 | 先检查是否有 Pod 未设置资源限制导致无限占用，或是否有大量已终止 Pod 未清理 |
| 直接重启节点 | 应查明压力来源，优先清理镜像、日志、调整 Pod 资源限制 |

## 3. Deployment 相关陷阱

### 3.1 滚动更新卡住
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 直接回滚而不查看具体原因 | 应先检查新 ReplicaSet 的 Pod 状态、事件、健康检查失败原因 |
| 忽略 Pod 中断预算（PDB） | 如果 PDB 的 `minAvailable` 过高，可能阻止足够 Pod 终止，导致滚动无法继续 |

### 3.2 回滚
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 回滚后不验证 | 回滚后应检查新版本是否正常、事件是否清除、业务是否恢复 |
| 误以为回滚会恢复 ConfigMap | 回滚只影响 Deployment 模板，ConfigMap 不会自动回滚，需单独处理 |

## 4. 网络相关陷阱

### 4.1 CNI 故障
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 重启 kubelet 或节点尝试恢复 | 应优先检查 CNI 插件 Pod 状态、日志、节点上配置文件 |
| 忽略节点防火墙 | 跨节点通信需要开放特定端口（如 VXLAN 4789、BGP 179），需检查安全组 |

### 4.2 Service 访问失败
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 直接检查 Service YAML | 应先确认 Endpoints 是否为空，再检查 kube-proxy 日志和 iptables/IPVS 规则 |
| 忽略 NetworkPolicy | 即使 Endpoints 正确，NetworkPolicy 可能拒绝流量，需检查策略 |

### 4.3 DNS 解析失败
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 直接重启 CoreDNS | 应先检查 CoreDNS 日志、Pod 状态、Service 是否正常、Pod 的 `/etc/resolv.conf` 配置 |

## 5. 控制平面故障陷阱

### 5.1 etcd 集群故障
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 直接重启 API Server | etcd 故障时 API Server 无法启动，应先恢复 etcd 多数节点或从快照恢复 |
| 没有快照就删除数据目录 | 始终先备份损坏的数据目录，再尝试恢复 |

### 5.2 API Server 证书过期
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 直接重启 API Server | 证书过期必须更新证书，否则重启无效 |
| 手动签发后不重启所有组件 | 证书更新后需重启 API Server、scheduler、controller-manager、kubelet 等，否则旧证书仍被缓存 |

### 5.3 Scheduler 故障
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 在修复 scheduler 前删除 Pending Pod | scheduler 修复后，Pending Pod 会自动尝试调度，无需删除 |
| 忽略自定义调度器名称 | 若 Pod 指定了非默认调度器，需确保该调度器正常运行 |

## 6. 通用安全陷阱

### 6.1 写操作
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 直接建议用户执行 `kubectl delete pod` 而不解释影响 | 应先说明该操作会触发重建，可能造成短暂中断，并确认用户是否允许 |
| 执行 `fix` 前未等待用户确认 | 必须获得用户明确同意（“是”、“确认”等），并使用 `--confirm` 标志 |

### 6.2 敏感信息泄露
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 输出完整 Pod 日志而不提醒 | 在输出前添加警告：“⚠️ Pod 日志可能包含敏感信息，请谨慎处理。” |
| 直接输出 Secret 内容 | 技能不会输出 Secret，但若日志或事件中包含，也应避免高亮显示 |

## 7. 诊断流程陷阱

| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 仅凭单一命令下结论 | 应综合 `sweep`、`pod`、`resources`、`events` 等多源数据交叉验证 |
| 跳过 `sweep` 直接深入单个 Pod | 可能遗漏全局性问题（如节点压力、CNI 故障），应先获取整体视图 |
| 忽略 RBAC 错误提示 | 遇到 RBAC 错误时，应引导用户查看 `SECURITY.md` 并配置最小权限，而不是尝试绕过 |

---

通过记录这些常见错误，模型可以在诊断时自我提醒，提高准确性和安全性。随着技能演进，可继续补充新的陷阱案例。
